# ABLab Quant — Инструкция

Покрываю каждую вкладку: **зачем, что вводить, как читать результат, типичные ошибки**.

## Быстрый старт за 60 секунд

1. Открой `src/index.html` в браузере (или хостинг-версию).
2. Вкладка **Data** → жми **Generate** в "Synthetic Generator" (по умолчанию n=5000, lift=2%).
3. Вкладка **Tests** → нажми **Run**. Увидишь t-test на сгенерированных данных.
4. Вкладка **CUPED / CUPAC** → жми **Run CUPED**. Сравни p-value до и после.
5. Вкладка **Report** → **Export Markdown** — все шаги выгрузятся.

Так ты прогонишь весь стек за минуту и поймёшь логику.

## 01 · Data — три способа загрузить данные

**CSV Upload.** Файл должен иметь колонки `group`, `metric`, опционально `pre` (для CUPED). Группы распознаются как `a`/`b`, `0`/`1`, `control`/`treatment`/`variant`. Разделитель определяется автоматически — переключи вручную, если парсер ошибся (точка с запятой в русских экспортах). Превью покажет первые 30 строк.

Пример: см. `examples/sample_experiment.csv`.

**Manual Summary.** Если у тебя только агрегированные числа из дашборда (mean / std / n) — введи их и нажми **Save as active**. Поля `pA`, `pB` опциональны и нужны только для z-prop / chi² на summary-уровне.

**Synthetic Generator.** Для AA-симуляций и обкатки тестов. Выбираешь распределение (Normal / Log-normal / Bernoulli / Poisson), `lift` — это процент эффекта, который вшивается в группу B. `Add pre-period covariate` создаёт коррелированную ковариату с ρ ≈ 0.5 — именно она нужна для CUPED.

После загрузки внизу появится блок **Active Dataset** — это сигнал, что данные подхватились и доступны остальным вкладкам.

**Типичная ошибка:** забыл нажать **Save as active** для manual summary → другие вкладки скажут «нет данных».

## 02 · Sample Size & MDE — сколько нужно юзеров

Три блока:

**Continuous Metric.** Вводишь baseline `μ` и `σ`, желаемый MDE в **абсолютных** единицах (если метрика = выручка в рублях, MDE = 50 ₽, не 5%), `α` (обычно 0.05) и power (обычно 0.8). Получаешь `n` на группу. Под капотом — `n = (σ·(z_{α/2}+z_β)/Δ)² · (1 + 1/k)`.

**Proportion Metric.** Для конверсий. MDE здесь **относительный** (0.02 = +2% к baseline). `p=0.1, MDE=0.02` означает «хочу поймать сдвиг с 10% до 10.2%».

**Reverse: MDE at given n.** Обратная задача — у тебя есть фиксированный трафик (например, 5000 юзеров за 2 недели), какой минимальный эффект ты сможешь поймать. Это правильный вопрос для slow-traffic экспериментов: часто ответ «никакой полезный».

**Power Curve.** После расчёта Continuous нажми **Draw from Continuous params** — увидишь, как power растёт с n. Красная линия = 0.8. Где кривая выходит на плато — нет смысла копить трафик дальше.

**Pro tip:** перепроверь себя — увеличь MDE вдвое, n должно упасть в ~4 раза (квадратичная зависимость).

## 03 · Tests — статистические тесты

**Source.** `Active dataset` использует raw-данные (CSV/синтетика). `Manual summary` — только summary-числа.

**Test choice — короткая шпаргалка:**

| Тест | Когда |
|---|---|
| Welch | дефолт для средних, разные дисперсии — берёт всегда, если не уверен |
| Student | если знаешь, что Var(A)=Var(B) — редко стоит выбирать |
| Mann-Whitney | непараметрика, тяжёлые хвосты, ranked metrics |
| z-prop | конверсии (бинарные) |
| chi² 2×2 | эквивалент z-prop, дает тот же p-value |
| Bootstrap | странные распределения, ratio-метрики, проверка чувствительности |

Bootstrap iterations — 5000 хватает для CI, для p-value — лучше 10000+.

**Чтение результата:** `t / z` — статистика, `p-value` — вероятность увидеть такое или большее различие при H₀, `diff` — наблюдаемый эффект, `se` — стандартная ошибка. Зелёная плашка — significant, красная — нет.

**Когда что-то не сходится:** если z-prop² ≠ χ² ровно — округление, не пугайся. Если MWU даёт значимость, а Welch — нет: у тебя skewed distribution (медиана сдвинулась, среднее — нет). Подумай, что тебе важнее как бизнес-метрика.

## 04 · Backtest — A/A валидация

**Зачем нужно:** проверить, что тест калиброван на твоих данных. Если эмпирический FPR при α=0.05 получается, например, 12% — тест систематически врёт (heavy tails, кластеризация, повторные наблюдения от одного юзера).

**Как использовать:**
1. Загрузи большой исторический датасет (без A/B сплита, просто метрика за период до эксперимента).
2. Выбери `metric column`.
3. `Iterations = 1000` — нормально для быстрого ответа, `5000` — для production-валидации.
4. **Run backtest.**

**Что смотреть:**
- **Empirical FPR** — должен быть близко к α (0.05) с учётом 95% CI. Зелёный — хорошо.
- **KS vs Uniform** — Колмогоров на p-values. Если < 0.05 — распределение p-values близко к равномерному (как и должно быть под H₀).
- **Гистограмма** — должна быть плоская, как красная пунктирная линия. Горб слева у нуля = тест слишком часто отклоняет H₀ (anti-conservative). Горб справа = слишком консервативный.

**Что делать, если FPR сломан:** переходи на bootstrap или MWU; проверь, нет ли в данных одного юзера с многими записями (нужно агрегировать до user-level либо использовать cluster bootstrap).

## 05 · Switchback — когда нельзя сплитить по юзерам

**Зачем:** marketplace-эффекты (price changes, dispatching algorithm), где A влияет на B через систему. Сплит по юзерам не работает — нужно чередовать по времени.

**Параметры:**
- **Горизонт** — длительность теста (часы).
- **Block size** — длина одного блока. Должна быть ≥ времени отклика системы. Для price changes 30–60 мин обычно норм; для матчинга — 15–30 мин.
- **Randomization:**
  - `Permuted blocks` — балансированные пары AB/BA. Лучший дефолт.
  - `Bernoulli` — каждый блок независимо. Может дать перекос, но проще анализировать.
  - `Latin square` — для day×hour-эффектов (хочешь покрыть все часы и дни в обеих группах).
- **Carryover buffer** — сколько минут в начале блока выкинуть из анализа (эффект A ещё «тянется» в начале B-блока).

**Что смотреть:** «Detectable MDE» — позеленело = твой ожидаемый эффект ловится с power 80%, покраснело = горизонта мало. Увеличь длительность или уменьши blocksize (если позволяет лаг системы).

**Важная оговорка:** реальные SE в switchback всегда выше теоретических из-за автокорреляции. CV в форме — твой грубый разгон под это; для production-анализа считай SE через **block-bootstrap** или **cluster-robust SE по дню** — иначе значимость будет завышена.

## 06 · CUPED / CUPAC — снижение дисперсии

**Зачем:** если у юзера есть pre-experiment метрика (выручка прошлого месяца, любая прокси), её можно использовать для контроля. Корректировка `Y' = Y − θ·(X − E[X])` уменьшает Var(Y') ≈ Var(Y)·(1 − ρ²). При ρ=0.7 — это **−49% дисперсии**, можно гнать тест на половине трафика.

**Как:**
1. Активный датасет должен содержать `metric`, `pre` (или другую ковариату), `group`.
2. Выбери колонки в селекторах (заполнятся автоматом, если назвал стандартно).
3. **Run CUPED.**

**Что читать:**
- **θ** — коэффициент регрессии (как сильно Y зависит от X).
- **ρ²(Y,X)** — это и есть твой ожидаемый процент сокращения дисперсии. Меньше 0.1 — CUPED почти ничего не даст, ищи лучшую ковариату.
- **Effective n boost** — во сколько раз меньше юзеров нужно для той же power.
- **Таблица raw vs CUPED** — p-value до и после. CUPED почти всегда даёт p ниже (или такой же).

**CUPAC vs CUPED:** в CUPAC ковариата X — это **прогноз ML-модели** на pre-period фичах. Делается отдельно: обучаешь GBDT/XGBoost на данных до эксперимента (target = твоя метрика, features = всё, что было до старта), сохраняешь предсказания как колонку `pre`, дальше всё то же. ABLab сам не учит модель — просто корректно использует X, который ты подложишь.

**Pitfall:** не считай θ отдельно для A и B — это вносит bias. Тулза считает на pooled, как и положено.

## 07 · Bayesian — без peeking-проблем

**Beta-Binomial (для конверсий).**
- `Prior α / β` — `1/1` (uniform prior, безопасный дефолт), `0.5/0.5` (Jeffreys), `s/n` (informed). Для большинства задач разница < 1%.
- Вводишь `successes / trials` для A и B.
- `Samples = 20000` — Monte Carlo для оценки P(B>A) и expected loss.

**Что читать:**
- **P(B > A)** — основная метрика. Принято решение: запустить B, если ≥ 95%.
- **E[lift]** — ожидаемый прирост.
- **E[loss | pick A/B]** — expected regret. Решающая метрика для бизнес-решений: «если выберу B и ошибусь, в среднем теряю X». Условие остановки: пока обе loss-функции выше threshold — продолжаем.

**Normal-Normal (для средних).** Аналогично, но для непрерывных метрик. `Prior τ₀ = 1e6` — практически плоский prior (рекомендуется, если нет prior knowledge). Получаешь P(B>A), credible interval разности и expected loss.

**Когда Bayesian лучше frequentist:** когда нужно прерывать тест досрочно, когда есть prior knowledge от прошлых экспериментов, когда стейкхолдеру важна интерпретация «вероятность, что B лучше», а не «p-value».

## 08 · SRM — must-have проверка

Запускай **каждый день** теста. χ²-тест, что фактический split совпадает с заданным.

**Ввод:** `10012, 9988` (наблюдаемые), `0.5, 0.5` (ожидаемые пропорции, сумма = 1). Работает для произвольного числа групп.

**Threshold не 0.05, а 0.0005.** Стандарт индустрии (Microsoft/Booking/Spotify): при стольких экспериментах false alarms на 5% сожгут команду. Зелёный — ок, жёлтый — копни, красный — **остановите тест и выясните причину**.

**Откуда берётся SRM в реальности:** баг в bucketing, потеря событий по одной из веток (frontend ошибка только в B), bot-фильтрация работает по-разному, кэширование сегментирует по cookie.

## 09 · Sequential (SPRT) — раннее завершение

**Зачем:** в фиксированном тесте подсматривать нельзя, иначе FPR → 1. SPRT/mSPRT решают: можно проверять каждый день и остановиться, как только LLR пересёк границу.

**Параметры:**
- `p₀` — null (статус-кво).
- `p₁` — alternative (минимальный эффект, ради которого затеялись).
- `α / β` — обычно 0.05 / 0.2.
- Текущие cumulative `successes / trials`.

**Decision rule** показан внизу:
- LLR ≥ upper → останавливай, есть эффект.
- LLR ≤ lower → останавливай, эффекта нет.
- В промежутке — продолжай.

**E[n | H1]** — сколько ожидать в среднем. Обычно ≈ 50% от fixed-horizon — это и есть выигрыш sequential подхода.

## 10 · Multi Testing — поправки

Если меришь 5+ метрик одновременно — нужны поправки, иначе хотя бы одна «значима» по случайности.

**Ввод:** список p-values через запятую или с переноса.

**Method:**
- **Bonferroni** — самая жёсткая, FWER. Используй для guardrail-метрик (где false positive дорог).
- **Holm** — uniformly лучше Bonferroni, тоже FWER. Дефолт, если нужен FWER.
- **BH (Benjamini-Hochberg)** — контроль FDR. Дефолт для exploration, когда метрик 10+ и допустимо 5% ложных открытий.
- **BY** — для зависимых метрик (когда метрики коррелируют между собой).

**Чтение:** колонка `p adj` — скорректированный p-value, его сравниваешь с α напрямую. ✓ — отвергаем H₀.

## 11 · Ratio Metrics — Delta method

**Зачем:** revenue per user, CTR per session, AOV — это `ΣX / ΣY`, а не среднее ratio. Простой t-test на user-level ratio даёт **смещённую** оценку и заниженный SE — false positives взлетают.

**Что вводить (по группам A и B):**
- `Σnum` — сумма числителя (общая выручка).
- `Σden` — сумма знаменателя (общее число сессий).
- `n` — число кластеров (юзеров).
- `Var(num)`, `Var(den)`, `Cov(num,den)` — на **user-level**. Считаются заранее в SQL/Python:
```sql
SELECT
  VARIANCE(user_revenue) AS var_num,
  VARIANCE(user_sessions) AS var_den,
  COVAR_SAMP(user_revenue, user_sessions) AS cov_nd
FROM user_aggregated
```

**Чтение:** SE учитывает covariance — это и есть фишка delta method. Сравни с наивным `t.test(R_a, R_b)` на той же выборке — обычно SE выше и значимость падает (правильно).

## Типичный workflow для нового эксперимента

1. **Sample Size** — посчитай нужный n до старта. Если трафика не хватает — обсуди с командой раньше, не позже.
2. **Backtest** на исторических данных — убедись, что тест калиброван (FPR ≈ α).
3. Запустил эксперимент в проде.
4. **SRM каждый день** — пока тест идёт.
5. После завершения: **Tests** (frequentist) **+ Bayesian** (для интуиции стейкхолдеров) **+ CUPED** (если есть pre-period).
6. Если несколько метрик — **Multi Testing**.
7. **Report → Export** → в ticket.
